Systems Engineering & UML Diagrams

Understanding systems, domain knowledge, and Unified Modelling Language

What Is a System?

A system is a set of interrelated components. Systems can be:

  • Physical or conceptual — a mobile phone vs. mathematical logic.
  • Natural or man-made — the human body vs. a sprinkler system.
  • Simple or complex — a door lock vs. a distributed banking platform.
Examples of Systems

The universe, a human body, mathematical logic, a mobile phone, a sprinkler system, and — critically — software applications themselves are all systems.

Core System Engineering Activities

Integration

The process of combining subsystems and components, then testing them to ensure they operate seamlessly together as a single, unified system. Integration is not just "putting pieces together" — it requires validating the interfaces and data flows between components.

Testing

Evaluating the fully integrated system to confirm that it meets all of its defined requirements and constraints. System-level testing looks at the whole, not individual parts.

Requirements Sub-Activities

The requirements phase itself usually consists of four sub-activities:

Sub-ActivityWhat It Does
ElicitationGathering needs from stakeholders through interviews, workshops, and observation.
AnalysisModelling and refining the gathered requirements to resolve conflicts and ambiguities.
SpecificationDocumenting the requirements formally (e.g., in an SRS document).
ValidationEnsuring the documented requirements actually reflect the client's real needs.

Constructing Analysis Models

The Role of Domain Knowledge

Development teams must deeply understand the specific field or "domain" they are building software for (e.g., banking, healthcare) to define accurate requirements. Domain knowledge forms the foundation for every subsequent phase: design, implementation, testing, and maintenance.

Why Modelling?

Modelling is heavily used during requirements elicitation to help teams structure their domain knowledge. It helps conceptualise and visualise the system, making it easier to analyse and communicate — especially to non-technical stakeholders.

Object-Oriented Perspective

In OO engineering, systems are specifically viewed as a collection of interacting objects. Each object has state (attributes) and behaviour (methods), and objects communicate by sending messages to each other.

UML Diagram Types

The Unified Modelling Language (UML) is the standard notation for representing software system designs. Different diagram types capture different aspects of the system.

DiagramWhat It ShowsKey Use
Class Diagram Static structure — classes, attributes, methods, relationships (inheritance, associations) Design-time blueprint; domain modelling
Use Case Diagram External entities (actors) and their interactions with the system's high-level functionality Requirements; scope definition
Sequence Diagram Object interactions in a specific scenario — the chronological order of messages exchanged over time Design; API contract documentation
State Diagram States an individual object can be in, and transitions between states triggered by events Modelling objects with complex lifecycles

Class Diagram — In Focus

Shows the static structure of a system. It includes the system's classes, their attributes, their methods (operations), and the relationships between them (like inheritance and associations). Domain models are a special case of class diagrams — they show only classes and relationships, without operations.

Example — Class Diagram (Shape hierarchy)

Use Case Diagram — In Focus

Illustrates how external entities (actors) interact with the system, highlighting the user's goals and the system's high-level functionality. Think of it as the "menu" of what the system can do.

Example — Use Case Diagram (Library System)

Sequence Diagram — In Focus

Represents object interactions within a specific scenario, mapping the chronological order of messages exchanged over time. Each vertical lifeline represents an object; horizontal arrows represent messages.

Example — Sequence Diagram (User Login Flow)

State Diagram — In Focus

Models the various states an individual object can be in, and the transitions between those states triggered by specific events. Useful for objects like orders, tickets, or user accounts that have distinct lifecycle phases.

Example — State Diagram (Order Lifecycle)

Structuring System Requirements

Hierarchical Breakdown

Requirements should be heavily detailed and structured hierarchically, using numbered notation to enable precise traceability:

R1 → R1.1 → R1.1.1 → R1.1.1.1 ...
Example Breakdown
  • R2.1 — Manage user profile (high-level)
  • R2.1.1 — Allow users to view and edit bio and profile picture
  • R2.1.2 — Enable users to switch between public and private profile settings

This type of structured numbering is the industry standard for a Software Requirements Specification (SRS) document, allowing engineers to easily trace specific features back to overarching business rules — and vice versa.

Why Systems Engineering Matters

Applying formal systems engineering principles provides four major benefits:

BenefitWhat It Means in Practice
Manages Complexity Clearly structures requirements, components, and interactions for full systems, rather than focusing on isolated code.
Improves Teamwork Forces coordination of roles, responsibilities, and system interfaces — exactly like real-world engineering teams do.
Strengthens Requirement Clarity Defining, refining, and validating requirements before building drastically reduces confusion and costly rework.
Prepares for Industry Real software projects rely on these exact principles for integration, testing, and deployment.
The Core Loop

Requirements → Model → Verify → Build → Integrate → Test. This loop, applied rigorously, is what separates professional software engineering from ad-hoc programming.
