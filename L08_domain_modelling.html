What Is a Model & Domain Modelling?

Model

A model is a conceptual representation or schematic description of a system, theory, or phenomenon that accounts for its properties and is used to study its characteristics.

Domain Modelling

Domain modelling is a process that helps a development team understand the specific application or domain they are working in, enabling them to establish a common understanding. It is driven by the language used in the requirements.

Why Domain Modelling Matters

Without a shared domain model, different team members build software based on different mental models of the problem space. This leads to integration failures, missed requirements, and costly rework.

UML Class Diagrams

A UML class diagram is a structural diagram that displays classes, their attributes, their operations, and the relationships between them.

Domain Model vs. Full Class Diagram

Crucial Distinction

A domain model is represented by a class diagram without operations. It shows only the structural concepts and relationships — no methods. Adding operations turns it into a full design-level class diagram.

Visual Structure of a Class

A class is represented by a rectangle divided into compartments:

CompartmentPositionContents
Class NameTopName of the class, usually capitalized (e.g., User).
AttributesMiddleData fields the class holds (e.g., name: String).
OperationsBottomMethods (omitted in a domain model).

Classes can be displayed in three views:

  • Compact: Class name only.
  • Semi-Compact: Name and selected attributes.
  • Expanded: All compartments fully shown.

Relationships & Multiplicity

Association

An association expresses a general relationship between two concepts that is distinct from inheritance or aggregation. These are often application-specific.

  • Example: "Instructor teaches Course", "User has Account".
  • Associations include an association name, a direction pointer, and the specific roles involved at each end.

Aggregation

Represents a "has-a" or "part-of" relationship. Used when possession expressions ("has", "possess") or containment expressions ("consist of", "part of") appear in the requirements.

Inheritance (Generalization)

Used when "X is a Y" expressions appear in the requirements. Represented by a solid line with a hollow arrowhead pointing to the parent class.

Multiplicity

Multiplicity defines the numerical constraints of a relationship. You must know these notations:

NotationMeaning
1Exactly one (the default).
1..*One or more.
0..1Zero or one (optional).
m..nFrom m to n.
* or 0..*Zero or more.
nExactly n.
Reading Multiplicity

Read multiplicities from the opposite end: the multiplicity next to class B tells you how many B's are associated with one instance of class A.

Example: "One base station services zero or more mobiles" → multiplicity of 0..* on the Mobile end.

The Brainstorming Rules

To build a domain model, the team analyses the language used in the requirements and maps parts of speech to UML concepts using these nine rules:

#Language PatternMaps To
1Nouns / noun phrasesClasses or attributes
2"X of Y" expressions (e.g., colour of car)X is an attribute of class Y
3Transitive verbsAssociation relationships
4AdjectivesAttribute values
5Numeric valuesAttribute or multiplicity values
6Possession expressions (has/have, possess)Aggregation or an attribute
7"Consist of / part of" expressionsAggregation relationships
8Containment / containing expressionsAssociation or aggregation
9"X is a Y" expressionsInheritance
Process Tip

Work through the requirements sentence by sentence. Highlight nouns in one colour and verbs in another before you start placing elements on the diagram.

ClickUP Class Exercise

The lecture provides this exercise as a practical walkthrough of building a domain model from requirements.

Requirements

  • FR1: A Student and Teaching Assistant can view announcements posted by their Lecturer.
  • FR2: Announcements can be deleted by an Administrator.

Model Solution

Applying the brainstorming rules to the requirements yields:

ElementTypeReasoning
UserClass (parent)Common noun; Student, Lecturer, Teaching Assistant, Administrator all ARE-A User → rule 9 (inheritance).
Student, Lecturer, Teaching Assistant, AdministratorClasses (children of User)Noun phrases; specialized roles → inheritance from User.
AnnouncementClassNoun phrase.
User views Announcement (0..*)AssociationTransitive verb "view" → rule 3.
Lecturer makes AnnouncementAssociationTransitive verb "post" → rule 3.
Administrator deletes AnnouncementAssociationTransitive verb "delete" → rule 3.
Operations in this Exercise

The lecture notes that while domain models should technically hide operations, the provided solution for this exercise does display operations such as login(), deleteAnnouncement(), and makeAnnouncement(). Be aware of this inconsistency when comparing to exam questions.

Key Takeaway

The power of a domain model lies in inheritance flattening: by having all roles inherit from User, you avoid duplicating the "views Announcement" association for every sub-role. One association at the parent level covers all children.
