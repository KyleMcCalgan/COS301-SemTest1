What Is Architectural Design?

The software architecture of a system refers to the style of design of its structure — including how its major components interface and interact. Architectural design is the decision-making process that determines this architecture for the system under development.

Why It Matters
  • Has significant impact on performance, efficiency, security, and maintainability.
  • Is the primary artifact for conceptualising, constructing, managing, and evolving the system.
  • Architectural design flaws may result in project failure.
  • Different types of systems require different design methods and architectural styles.

Design Considerations

Before choosing a style, consider: ease of change and maintenance; use of COTS (Commercial Off-The-Shelf) parts; system performance (real-time data, high-volume transactions); reliability; security; software fault tolerance; recovery.

The Design Process

Steps in Order
  1. Determine design objectives — performance, reliability, security, etc.
  2. Determine the type of system — Interactive, Event-Driven, Transformational, or Persistence.
  3. Apply an architectural style — or perform custom design if no existing style fits.
  4. Specify subsystem functions, interfaces, and interaction behaviour.
  5. Review the architectural design — against requirements, principles, and security.

Four System Types

TypeDescriptionKey CharacteristicsArchitectural Style
Interactive Fixed sequence of actor requests and system responses Client-server relationship; actor initiates and ends interaction; state reflects business process progress N-Tier
Event-Driven Responds to randomly arriving events; state-dependent responses No fixed sequence; interacts with multiple external entities simultaneously; may have timing constraints Event-Driven
Transformational Network of information-processing activities; batch process Little or no interaction during transformation; usually stateless; may be computation-intensive Main Program & Subroutines
Object-Persistence Provides object storage and retrieval; hides implementation Responsible only for storing/retrieving objects; does little or no business processing Persistence Framework
Recursive View

Systems and subsystems are relative to each other — a system is a subsystem of a larger system; a subsystem is a system that may consist of other subsystems. A system may contain different types of subsystems, each requiring a different design method.

Architectural Styles

N-Tier Architecture

Organises the system into distinct layers. Each layer has a high-level responsibility and communicates only with adjacent layers.

User → [GUI Layer] → [Controller Layer] → [Business Objects Layer] → [Database Layer]

Represented in UML as a package diagram with <<import>> dependencies (dashed arrows) between layers. Packages contain their public (+) and private (−) components.

Client-Server Architecture

Multiple clients communicate with a shared server via remote procedure calls. Clients provide access to humans or systems; the server (commonly a database server) processes requests. Use for: distributed systems where multiple users access shared resources.

Main Program and Subroutine Architecture

A main program calls subroutines hierarchically. Straightforward to understand and trace. Use for: batch processing, transformational systems.

Event-Driven Architecture

A state-based controller receives events from external entities and directs multiple components. Responses depend on current state, not a fixed sequence. Use for: control systems, real-time systems, GUIs.

Object-Persistence (Database) Framework

A DB Manager mediates between business objects (in an OO language) and multiple databases (using DBMS-specific languages), hiding database differences from the rest of the system. Business objects do not need to know what database is used.

Custom Architectural Design

Not all application systems fit an existing architectural style. Custom architectural design is required to meet specific application needs. Design patterns are a key tool for custom design.

Specifying Subsystems

Once an architectural style is chosen or a custom design is created:

  1. Allocate requirements and design objectives to subsystems and components.
  2. Specify the functionality of each subsystem.
  3. Specify the interfaces between subsystems.
  4. Specify the interaction behaviour between subsystems.

Software Design Principles

PrincipleDescription
Design for ChangeBuild in a mechanism to adapt to anticipated changes — volatility should be isolated
Separation of ConcernsFocus on one aspect of the problem in isolation — don't mix concerns in one module
Information HidingShield implementation details to reduce change impact — only expose what is necessary
High CohesionHigh relevance of all a module's functions to its core purpose — do one thing well
Low CouplingReduce the run-time effect and change impact of one subsystem on others
KISSKeep It Simple and Stupid — design simple, focused "stupid objects"; avoid unnecessary complexity

Review & Guidelines

Reviewing the Design

  • Does the design meet requirements and architectural objectives?
  • Does it satisfy software design principles?
  • Does it satisfy security requirements and secure design principles?

Guidelines for Good Architectural Design

  1. Adapt an existing architectural style when possible.
  2. Apply software design principles throughout.
  3. Apply design patterns where appropriate.
  4. Check against design objectives and principles.
  5. Iterate the steps if needed.
Agile Perspective
  • Value working software over comprehensive documentation.
  • Apply the 20/80 rule — good enough is enough. Do not over-engineer the architecture.
