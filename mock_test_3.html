Q1 Single MCQ 2 marks

Which statement most accurately describes a domain model in UML?

▶ Show Answer
✓ Correct Answer: A
✓ A: A domain model captures the conceptual entities in the problem domain — what things exist, their attributes, and how they relate. It intentionally omits operations and implementation detail to stay at the analysis level.
✗ B: Although domain models can inform database design, they are not schemas. Domain models show conceptual entities; schemas show tables, columns, and constraints.
✗ C: Static structure only. Dynamic behaviour is shown in sequence diagrams, activity diagrams, or state machine diagrams.
✗ D: An implementation class diagram includes operations, visibility (+/-/#), data types, and constructor signatures. A domain model omits all of these — it is intentionally abstract.
Domain model ≠ implementation class diagram. The domain model is an analysis artefact used to understand the problem. The class diagram is a design artefact used to guide implementation. The domain model informs but does not equal the class diagram.

Q2 Multi-Select MCQ 4 marks

A ride-hailing platform uses a queue-based dispatch system with these design decisions:

  • API responds "driver assigned" as soon as the ride request enters the queue
  • Driver locations are cached for 60 seconds
  • Failed queue messages are retried 10 times immediately with no backoff
  • The Ride-State DB uses async replication with no idempotency keys

Select ALL options that correctly identify design flaws. (Negative marking applies.)

⚠ Negative Marking — each incorrect selection deducts marks.
▶ Show Answer
✓ Correct: A, B, D, F
✓ A — No idempotency keys: If a worker crashes mid-processing, the message is re-queued and processed again. Without idempotency keys, the operation runs twice — creating duplicates. This is a fundamental queue-based design requirement.
✓ B — Premature "assigned" response: Placing a request in a queue is not the same as securing a driver. The user is told an assignment happened when it hasn't yet — misleading and problematic if the queue processing fails.
✓ D — Retry storm: 10 immediate retries means a single failing message generates 10 rapid requests to the downstream service. Under degraded conditions this amplifies load — the pattern should use exponential backoff with jitter.
✓ F — Cache coherence bug: Two workers reading from the same 60-second cache can both see driver X as available. Both assign driver X. Result: one driver, two riders. The assignment must lock or use compare-and-swap at commit time.
✗ C — FIFO does NOT prevent duplicate assignment: FIFO orders messages but cannot prevent concurrent workers from both reading the same driver-location cache entry and assigning independently. Concurrency control requires locking, not just ordering.
✗ E — Async replication does NOT block the primary: That is the definition of SYNCHRONOUS replication. Async means the primary commits and continues immediately; replicas catch up later. Async replication reduces write latency at the cost of potential data lag.
Queue-based design anti-patterns to memorise: (1) no idempotency keys → duplicates; (2) premature success response → misleading state; (3) no backoff → retry storms; (4) stale cache → race conditions at assignment.

Software engineering is organised into three parallel tracks. Fill in the blanks.

Word bank: quality assurance, project management, software development, testing, deployment

1. The first track, , encompasses requirements engineering, design, implementation, and all activities that directly produce the software artefacts.

2. The second track, , ensures that software artefacts meet defined requirements and quality standards throughout the lifecycle — not just at the end.

3. The third track, , covers planning, scheduling, risk management, resource allocation, and team coordination to ensure the project is delivered on time and within budget.

▶ Show Answer
✓ 1 → software development   2 → quality assurance   3 → project management
Software Development: The primary production track — directly building the system. Includes requirements, design, coding, unit testing.
Quality Assurance (SQA): A meta-level track that monitors and verifies the quality of artefacts and processes across the entire project, not just the code. Includes audits, reviews, and standards compliance.
Project Management: The coordination track — ensuring the right resources are in the right place at the right time. Handles risk, schedule, and budget.
These three tracks run in parallel throughout the project. QA is often misunderstood as synonymous with "testing" — but QA is broader. It includes process audits and artefact reviews, ensuring quality is built in rather than inspected in at the end.

Q4 Matching 4 marks (1 per match)

Match each Scrum ceremony to its correct description.

⚠ Negative Marking — each incorrect match deducts marks.
Sprint Planning
Daily Scrum
Sprint Review
Sprint Retrospective
▶ Show Answer
✓ Sprint Planning → "What/How"; Daily Scrum → 15-min inspect/adapt; Review → demo to stakeholders; Retrospective → process improvement
Sprint Planning (timebox: up to 8h for 1-month sprint): Product Owner presents priorities. Developers select items and create the Sprint Goal. Part 2: Developers plan how to deliver the selected work.
Daily Scrum (timebox: 15 min): Owned by Developers. Not a status report. Three questions are a starting point: what did I do, what will I do, any impediments?
Sprint Review (timebox: up to 4h for 1-month sprint): The team shows working software to stakeholders. The Product Backlog may be adjusted based on feedback. It is an inspect-and-adapt event on the product.
Sprint Retrospective (timebox: up to 3h for 1-month sprint): Internal team event. Focus: process, relationships, and tools — not the product. Produces action items for the next sprint. It is an inspect-and-adapt event on the team's way of working.
Order: Planning → Daily Scrum (throughout) → Review → Retrospective → next Planning. Review = inspect the product; Retrospective = inspect the team's process. Both happen at the end of each sprint.

Q5 Fill in the Blank 2 marks (0.5 per blank)

A student is drawing a UML use case diagram for an online library system. Fill in the blanks using correct UML terminology.

Blank 1: The represents a person, organisation, or external system that interacts with the system to achieve a goal.
Blank 2: Each oval in the use case diagram represents a — a user-observable goal the actor wants to achieve.
Blank 3: When "Borrow Book" always executes "Update Loan Record" as part of its flow, the relationship from "Borrow Book" to "Update Loan Record" is an relationship.
Blank 4: When "Search Catalogue" may optionally trigger "Send Recommendation Email" under certain conditions, that optional extension is an relationship.
▶ Show Answer
✓ Blank 1: actor  | Blank 2: use case  | Blank 3: include  | Blank 4: extend
Actor: Drawn as a stick figure (person) or rectangle (external system). Actors are outside the system boundary.
Use Case (oval): A named, user-observable goal. The name should be a verb phrase: "Borrow Book", "Search Catalogue", "Return Book".
<<include>>: Always executes the included use case. Modelled as a dashed arrow from the base use case to the included use case, labelled <<include>>.
<<extend>>: Optional extension that occurs under specific conditions. The extension point is defined in the base use case. Dashed arrow points FROM the extending use case TO the base.
Memory hook: include = mandatory (it always happens); extend = optional (it extends under a condition). The arrow direction differs: include arrow points TO the sub-use-case; extend arrow points TO the base use case being extended.

Q6 Single MCQ 2 marks

A municipality hires a software team to build a system for managing water outages. Every stakeholder has conflicting priorities, requirements keep shifting as the team learns more about the domain, there is no single "correct" solution, and every attempt at a solution changes the understanding of the problem.

This project exhibits the characteristics of a:

▶ Show Answer
✓ Correct Answer: B — Wicked problem
✓ B: Rittel and Webber's wicked problem characteristics all appear: ill-defined requirements, no definitive solution, every solution attempt changes the problem, stakeholder conflicts, and no "stop" rule. Software engineering deals with wicked problems — which is why rigid Waterfall fails for most real projects.
✗ A (Tame): A tame problem is well-defined with a known solution approach (e.g., "implement a sorting algorithm"). It may be difficult but the problem itself is understood.
✗ C, D: Spiral and Waterfall are process models, not problem classifications. The tame/wicked distinction describes the nature of the problem, not the methodology for solving it.
The tame/wicked distinction justifies iterative development. If software problems were tame, Waterfall would work perfectly — you'd gather all requirements, build, and ship. Because requirements are wicked (they evolve as understanding grows), iterative/Agile approaches that embrace change are more successful.

Q7 Multi-Select MCQ 5 marks

A student team demonstrates their Smart Campus app to examiners. During the demo: the app takes 12 seconds to load a map (they blame it on the venue WiFi); sensor readings are visibly wrong; one presenter says "it works on my machine"; they cannot answer "how does this scale to 10,000 users?"; and they skip the notification feature saying "we ran out of time."

Select ALL statements that correctly analyse the failures of this demo. (Negative marking applies.)

⚠ Negative Marking — each incorrect selection deducts marks.
▶ Show Answer
✓ Correct: B, C, E, F, G
✓ B — Poor preparation: Production-grade demos must be tested in the target environment (or against a local mock). Blaming venue WiFi is a preparation failure.
✓ C — Architecture knowledge gap: Every team member presenting should be able to explain scalability, technology choices, and trade-offs. Not knowing your own architecture is a critical presentation failure.
✓ E — Integration issues: "Works on my machine" is the classic symptom of environment-specific configuration, missing dependencies, or unmerged code paths. Demo environments must mirror the tested environment.
✓ F — Performance issue: 12 seconds is objectively a performance problem. Blaming the network is not an analysis — it is an excuse. The team should have cached the map tiles or tested offline performance.
✓ G — Incomplete demo / insufficient rehearsal: Skipping a feature reveals either it doesn't work or the team didn't allocate enough time. Both reflect poorly on preparation and project completion status.
✗ A: Blaming venue network is not professional handling — it is deflection. Professional handling would be switching to pre-recorded video, local server, or offline mode.
✗ D: Software teams are responsible for graceful handling of hardware errors — showing "N/A" or "sensor offline" is better than displaying wrong values.
✗ H: Wrong readings are a bug, not an adaptive feature.
Demo preparation rule: "Never demo code you haven't run in the demo environment." Run end-to-end in the exact setup (laptop, network, projector) at least once before the actual demo.

Q8 Matching 4 marks (1 per match)

A system architect is decomposing a large home automation project into subsystems. Match each decomposition example to the strategy it represents.

⚠ Negative Marking — each incorrect match deducts marks.
The project is split into a "Hardware Integration" workstream, a "Mobile App" workstream, and a "Cloud Backend" workstream — each handled by a specialist team.
The system is divided into "Motion Detection," "Climate Control," and "Security Alerts" subsystems — each corresponding to a distinct requirement cluster.
The team re-uses the Zigbee communication protocol from the client's previous home automation system as the backbone, structuring new components around it.
The system components are structured to mirror the client company's departments: "Installation Team," "Support Desk," and "Billing" each get their own module.
▶ Show Answer
✓ Row 1 → By Discipline   Row 2 → By System Functions   Row 3 → By Existing Architecture   Row 4 → By Functional Units of the Org
By Discipline: Separates work by the type of expertise required (hardware, software, UX, QA). Team structure drives system structure.
By System Functions: Divides by what the system does — each subsystem corresponds to a distinct functional requirement cluster.
By Existing Architecture: Leverages constraints or assets already in place (legacy protocols, existing codebase). Pragmatic — avoids reinventing the wheel but can constrain design choices.
By Functional Units of the Organisation: Aligns system boundaries with the client's org chart — each department gets its own module. Risk: if the org restructures, so must the system (Conway's Law).
Conway's Law: "Organizations design systems that mirror their own communication structure." Decomposing by organisational units is natural but can create long-term coupling between system architecture and org structure.

Q9 Jumbled Sentence 4 marks (1 per blank)

Match each SDLC activity description to the correct phase name from the word bank.

Word bank: Requirements, Design, Implementation, Testing, Maintenance

1. The team interviews stakeholders, writes user stories, and produces a Software Requirements Specification (SRS) document. This is the phase.

2. The architect produces the system architecture diagram, the database schema, and the interface specifications. This is the phase.

3. Developers write code, commit to version control, and run automated builds on the CI server. This is the phase.

4. After production deployment, the team applies security patches, fixes reported bugs, and adds small features requested by users. This is the phase.

▶ Show Answer
✓ 1 → Requirements   2 → Design   3 → Implementation   4 → Maintenance
Requirements: Answers "what should the system do?" — elicitation, analysis, specification, and validation of requirements. Artefact: SRS.
Design: Answers "how will the system be built?" — architectural decisions, component design, data modelling. Artefact: design documents, diagrams.
Implementation: Writing code and integrating components. Artefact: source code, executable.
Testing: (not shown above) Verifying the implementation against requirements — unit, integration, system, UAT.
Maintenance: Post-deployment. Corrective (bug fixes), adaptive (env changes), perfective (feature additions), preventive (refactoring). Typically the longest and most costly phase over the system's lifetime.
A key fact: Maintenance typically consumes 60–80% of total software lifecycle cost. This is why design quality (maintainability, modularity) has such an enormous ROI — small improvements in code quality compound massively over years of maintenance.
