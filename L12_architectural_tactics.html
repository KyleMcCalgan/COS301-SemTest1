What Is a Tactic?

An architectural tactic is a specific design decision that influences how a system responds with respect to a quality attribute. Tactics are smaller than a full architecture, but powerful enough to materially change system behaviour.

Core Mental Model
LayerRole
Architectural styleProvides structure
Architectural tacticsShape behaviour
Trade-offsDetermine whether the design is acceptable
Critical Point

Choosing an architectural style does not automatically guarantee any quality attribute. An N-tier system is not automatically scalable. An event-driven system is not automatically reliable. The actual quality response depends on the tactics used inside the architecture.

Scalability & Performance Tactics

Scale-Up Resources

Increase processing power, communication channel capacity, or storage on existing hardware.

Scale-Out Resources

Add more instances: service provider clustering, grids, external resources. Distribute tasks across specialised or commoditised resources.

Reduce Load — Spread & Optimise

CategoryTactics
Spread load across resourcesClient-driven / controlled / server-driven load balancing
Spread load across timeScheduling, queueing, concurrency, pre-processing
Efficient communicationOptimise protocol, compress messages, batch requests, coarse-grained calls
Efficient persistenceIndexing, optimise persistence structure, optimise queries
Efficient storagePaging, compression, ordering/defragmentation
Resource re-useThread pooling, connection pooling, object pooling, caching

Reliability Tactics

GoalTactics
Prevent Faults Resource locking, removal from service, messaging service, active redundancy, contracts-based design, testing framework, remove single points of failure
Detect Faults Error/exception communication, resource monitoring, checkpoint evaluation, deadlock detection, message integrity, logging, service provider oversight, self-testing
Recover from Faults Fix problem (RAID, clustering, disaster recovery), passive redundancy, reset service provider, root cause determination, checkpoint rollback, maintain backups

Security Tactics

GoalTactics
Resist Attacks Limit access: minimise access channels & domain, authentication, authorisation, confidentiality, protect in depth.
Limit exposure: ring-fence resources, additional layers for valuables, treat external resources as untrusted, fail securely, enforce secure defaults, minimise complexity.
Detect Attacks Determine message integrity, event logging and analysis, scan for attack signatures, audit and monitor sensitive events
Recover from Attacks Drop connection / drop requests, update access rules, restore state

Quality Attributes in Tension

Architectural tactics rarely improve only one quality for free. Architects must make defensible trade-offs.

TacticBenefitCost
CachingImproves performanceCan weaken data consistency
ReplicationImproves availabilityCan complicate correctness; replication lag
EncryptionImproves securityAdds computational overhead; usability friction
QueuingImproves reliability under burstsIncreases latency; complicates consistency
Additional abstractionImproves modifiabilityReduces simplicity; can reduce performance
Architectural design is a trade-off activity, not a feature checklist. Architects do not search for perfect solutions — they make defensible trade-offs under constraints.

Tactic Deep Dives

Caching

Stores previously computed or retrieved results so the system avoids repeated expensive work. Useful when: the same data is read many times; re-fetching cost is high; slightly stale data is acceptable.

Caching Decision Guide
Data ItemDecisionReasoning
Marketing banner imagesCache aggressivelyRead-heavy, rarely changes, staleness acceptable
Event descriptions & venue detailsCache cautiouslyChanges occasionally; use a short TTL
Seat availability (nearly sold out)Avoid cachingHighly volatile; stale data → overselling
Payment authorisation responsesAvoid cachingMust be fresh and authoritative; correctness critical
Hidden Caching Trade-offs
  • Stale data can be served to users
  • Invalidation rules are often hard to define correctly
  • Cold starts can create sudden load spikes
  • Memory pressure and eviction behaviour affect predictability
  • Different parts of the system may observe different values simultaneously

Replication & Load Balancing

Running more than one instance of a component so failure of one does not stop service. A load balancer distributes requests across healthy replicas. Must decide: how much inconsistency is acceptable and how quickly must replicas converge?

Costs: Write conflicts, replication lag, inconsistent reads, operational complexity, harder debugging and incident analysis.

Queuing

Allows one part of the system to accept work quickly and defer processing. Useful when demand arrives in bursts, work can be processed asynchronously, and temporary delay is acceptable.

Engineering Cost of Queuing
  • Increased end-to-end latency
  • Duplicate message delivery
  • Out-of-order processing
  • Poison messages that repeatedly fail
  • Backlog growth during downstream slowdown
  • Eventual consistency — "accepted" ≠ "confirmed"

Circuit Breakers

In distributed systems, one failing dependency can trigger cascading failures. A circuit breaker stops repeated calls to a failing dependency for some period, allowing the rest of the system to degrade gracefully. Requires clear fallback behaviour and careful monitoring.

Modifiability: Isolate Responsibilities

  • Isolate volatile parts of the system
  • Depend on abstractions rather than concrete implementations
  • Use interfaces or contracts at component boundaries
  • Localise change so one modification does not ripple across the system
Selective Architecture

The best architecture is often not uniform. It is selective.

Action TypeAppropriate Tactic
Browsing catalogue dataAggressive caching — read-heavy, tolerant of staleness
Search & recommendationsAsynchronous processing, optimised for speed
Final booking / payment / inventoryStrict consistency, no caching, synchronous confirmation

Architectural Patterns

Layered Pattern

Components organised in layers; each layer accesses only the adjacent lower layer. Benefits: Pluggable and replaceable layers; improved cohesion; complexity reduction; loose high-level coupling; ability to mock lower layers; layers can be developed by different teams.

Client-Server (2-Layer)

Client application provides access to humans or systems. Server is commonly a database with stored procedures.

N-Layer

Multiple specialised layers: access layer → processing layer → infrastructure layer → persistence layer.

Pipes & Filters

  • Filters: stateless services or pure functions — state does not survive across calls; independent of other filters.
  • Pipes: input/output connectors between filters.

Classic Unix example: ls | head -4 | tail -2 > output.txt

How to Reason About Tactics

Use this framework in exams and in practice. Work through these questions in order:

Tactic Reasoning Framework
  1. What quality attribute am I trying to improve?
  2. What exact problem am I observing?
  3. What workload or failure shape causes that problem?
  4. What tactic is appropriate for that specific context?
  5. What trade-offs or new risks does that tactic introduce?
  6. How will the system behave if the tactic itself fails?
A strong architecture answer does not stop at naming a style. It explains the problem, the tactic, why it fits, the trade-offs, and how the resulting risks will be managed. That is the level of thinking expected from an engineer.
