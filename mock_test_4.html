Q1 Jumbled Sentence 4 marks (1 per blank)

Fill in the blanks to correctly sequence and describe the Scrum sprint lifecycle.

Word bank: Sprint Planning, Daily Scrum, Sprint Review, Sprint Retrospective

1. Every sprint begins with , where the team selects work from the Product Backlog, agrees on a Sprint Goal, and creates the Sprint Backlog.

2. Each working day of the sprint, the Developers hold a — a 15-minute inspection and adaptation of the team's progress toward the Sprint Goal.

3. At the end of the sprint, the team demonstrates working software to stakeholders in the , inspecting the increment and adapting the Product Backlog based on feedback.

4. Before beginning the next sprint, the team reflects on their ways of working in the and identifies concrete process improvements.

▶ Show Answer
✓ 1 → Sprint Planning   2 → Daily Scrum   3 → Sprint Review   4 → Sprint Retrospective
Sprint Planning starts the sprint. Two parts: (1) What — select backlog items and set Sprint Goal; (2) How — Developers decompose into tasks.
Daily Scrum is the inspect-and-adapt heartbeat during the sprint. Owned by Developers. Not a status report. Time-boxed to 15 minutes.
Sprint Review = inspect the product. Stakeholders attend. The increment is demonstrated. Feedback adjusts the Product Backlog. It is NOT a sign-off meeting.
Sprint Retrospective = inspect the team's process. Internal. Focus: what went well, what to improve, and commitment to one actionable change next sprint.
Common mistake: confusing Sprint Review and Sprint Retrospective. Review = product focus, external stakeholders invited. Retrospective = team/process focus, internal only.

Q2 Single MCQ 2 marks

An architect says: "We need to add a circuit breaker between the OrderService and PaymentService so that if PaymentService is slow, it doesn't block OrderService threads indefinitely."

In the vocabulary of software architecture, what has the architect specified?

▶ Show Answer
✓ Correct Answer: B — Architectural tactic
✓ B (Tactic): A tactic is a focused design decision that directly improves a specific quality attribute (here: reliability/availability). The circuit breaker is a well-known reliability tactic — it detects failure and temporarily stops calls to the failing service, preventing cascading failures.
✗ A (Style): A style describes the macro-level structure (Layered, Microservices, Event-Driven). "Add a circuit breaker" is too specific and localised to be a style.
✗ C (Use case): Use cases capture actor goals. The circuit breaker is an internal design mechanism, invisible to actors.
✗ D (Functional requirement): Functional requirements define what the system must do (verifiable behaviour). The circuit breaker is an architectural quality mechanism — a non-functional concern.
Tactics vs. Styles: Styles answer "how is the system structured at large?" Tactics answer "how do we achieve a specific quality attribute?" A single architectural style can employ many tactics. Example: a Microservices style commonly uses circuit breakers (reliability), rate limiters (performance), and service registries (modifiability) as tactics.

Q3 Multi-Select MCQ 4 marks

A developer starts on branch feature/search. The working directory contains search.py (modified) and config.py (modified). The staging area is empty. The developer runs these commands in order:

  1. git add search.py
  2. git commit -m "add search logic"
  3. git stash

Select ALL statements that correctly describe the state after all three commands complete. (Negative marking applies.)

⚠ Negative Marking — each incorrect selection deducts marks.
▶ Show Answer
✓ Correct: A, C, E, G
✓ A: git add search.py staged only search.py. The commit packages whatever is staged — so only search.py changes are in the commit.
✓ C: After the commit, the staging area is empty. config.py is still modified in the working directory. git stash takes all tracked modified files (including unstaged) and saves them. After stash, the working directory is clean.
✓ E: config.py was never git add-ed, so it was never staged. After the commit, it remains in a "modified but unstaged" state.
✓ G: git stash is a temporary stack — it stores your work-in-progress. git stash pop (or git stash apply) restores the stashed changes.
✗ B: config.py was NEVER staged — it remained in the working directory, not the staging area.
✗ D: git stash does not commit — it saves to a stash stack and reverts the working directory. No new commit is created.
✗ F: Stash is reversible. The changes are safely stored and can be reapplied.
Stash is useful when you need a clean working directory (e.g., to git pull or switch branches) without committing work-in-progress. It stashes tracked modified files by default; use git stash -u to also stash untracked files.

Q4 Matching 4 marks (1 per match)

Match each security tactic to the threat or scenario it addresses.

⚠ Negative Marking — each incorrect match deducts marks.
A healthcare API requires a valid JWT token signed with RS256 in the Authorization header. Requests without a valid token receive HTTP 401.
Even after authenticating, a junior doctor can view patient records but cannot delete them or access billing information — that is restricted to administrators.
Patient data is stored in AES-256 encrypted columns. Even if an attacker obtains the database file, the data is unreadable without the encryption key.
The API only exposes the minimum fields needed per endpoint. Patient SSN is never returned in list responses — only in the specific "Get Patient Detail" endpoint accessible to authorised roles.
▶ Show Answer
✓ Row 1 → Authenticate Actors   Row 2 → Authorise Actors   Row 3 → Encrypt Data   Row 4 → Limit Exposure
Authenticate Actors: Verifies identity — "who are you?" Mechanisms: JWT, OAuth, API keys, passwords + MFA.
Authorise Actors: Controls what an authenticated actor can do — "what are you allowed to do?" Mechanisms: RBAC (Role-Based Access Control), ABAC, permission scopes.
Encrypt Data: Protects data confidentiality at rest and in transit. Even if an attacker accesses the storage layer, encrypted data is useless without the key.
Limit Exposure: Reduce the attack surface by exposing only what is necessary. Principle of least privilege applied at the data/API level. Complementary to authorisation.
Security is layered: Authenticate (who?) → Authorise (what can they do?) → Encrypt (protect data) → Limit Exposure (reduce what can be exploited). Each layer defends against different attack vectors.

Q6 Single MCQ 2 marks

A QA engineer says: "We need to ensure the system does what the specification says it should do." A product manager says: "We need to ensure the system actually solves the user's real problem."

Which statement correctly maps these two concerns to their testing counterparts?

▶ Show Answer
✓ Correct Answer: C
✓ C: The classic V&V distinction: Verification = "are we building the product right?" — checks the system against its specification. Validation = "are we building the right product?" — checks the system meets actual user needs.
✗ A: Only the QA engineer describes verification. The product manager is asking about real-world user needs — that is validation.
✗ B: Only the product manager describes validation. The QA engineer is checking against the spec — that is verification.
✗ D: These are reversed. The spec-checking activity is verification, not validation.
Memory hook: Verification = internal document review, inspections, static analysis, unit tests against spec. Validation = user acceptance testing (UAT), usability studies, customer sign-off. You can pass verification (spec is met) but fail validation (the spec was wrong).

Q7 Matching 4 marks (1 per match)

Match each description to the correct software engineering term or methodology.

⚠ Negative Marking — each incorrect match deducts marks.
A standardised visual modelling language used to specify, visualise, construct, and document the artefacts of object-oriented software systems.
A set of planned and systematic activities that ensure software artefacts conform to requirements, standards, and quality goals throughout the development lifecycle.
An iterative software process framework that organises development into Inception, Elaboration, Construction, and Transition phases, driven by use cases and centred on architecture.
A specialisation of software engineering that views the world as interacting objects with encapsulated state and behaviour — use cases are used to capture requirements.
▶ Show Answer
✓ Row 1 → UML   Row 2 → SQA   Row 3 → RUP   Row 4 → OOSE
UML (Unified Modelling Language): Created by Booch, Rumbaugh, and Jacobson. A visual language, not a process. Diagrams include use cases, class diagrams, sequence diagrams, state machines, and more.
SQA (Software Quality Assurance): A cross-cutting concern that spans all phases. Includes code reviews, inspections, audits, and standards compliance checks — not just testing.
RUP (Rational Unified Process): Iterative but heavyweight. Customisable — teams select which RUP artefacts to produce. Key characteristics: use-case driven, architecture-centric, iterative and incremental.
OOSE (Object-Oriented Software Engineering): Ivar Jacobson's methodology. Introduced use cases as the primary mechanism for capturing requirements and driving design. Objects are the central abstraction.
Common CT trap: SQA and RUP descriptions are often swapped. Remember — SQA is about quality assurance processes and standards (broad, lifecycle-spanning). RUP is a specific process model with four phases and six disciplines.

Q8 Multi-Select MCQ 4 marks

An architect is adding scalability tactics to a high-traffic e-commerce product catalogue service. Select ALL options that represent valid scalability tactics. (Negative marking applies.)

⚠ Negative Marking — each incorrect selection deducts marks.
▶ Show Answer
✓ Correct: A, C, D, F
✓ A — Read Replicas: Horizontal scaling of read capacity. Write traffic still goes to the primary, but read queries are distributed across replicas. Classic scalability pattern.
✓ C — Caching: Reduces load on the database by serving frequently-read data from fast in-memory storage. TTL controls staleness. A core performance/scalability tactic.
✓ D — Sharding: Horizontal partitioning of data. Each shard handles a subset of the data, allowing the write throughput and storage capacity to scale with the number of shards.
✓ F — CDN: Offloads static asset delivery from the origin server to geographically distributed edge nodes. Reduces latency for users and eliminates origin server bandwidth for static content.
✗ B — Single monolith (vertical only): Vertical scaling has a hard ceiling (the largest available machine). It is not a scalability tactic — it is a stopgap. The question asks for scalability tactics, implying sustainable horizontal scale.
✗ E — Encryption: A security tactic, not a scalability tactic. Encrypting columns adds CPU overhead and can slightly reduce throughput — the opposite of a scalability improvement.
✗ G — Synchronous logging in critical path: This is an antipattern for scalability — synchronous I/O in the request path adds latency and reduces throughput. Logging should be asynchronous.

Q9 Jumbled Sentence 4 marks (1 per blank)

A test team is choosing an integration testing strategy for each project scenario. Select the best-fit strategy from the word bank.

Word bank: Big Bang, Top-Down, Bottom-Up, Sandwich

1. The team needs to test the user-facing navigation flow early, before lower-level data services are ready. They create stub responses for the database layer. Strategy:

2. The team's shared utility libraries (logging, authentication, data access) are ready first. They write driver programs that call these libraries and verify them independently before building higher-level components. Strategy:

3. A startup with a tight deadline integrates all 12 modules on the final day of development and runs a full system test — hoping it all works together. Strategy:

4. A large banking system simultaneously tests the UI workflow from the top using stubs and tests the transaction processing layer from the bottom using drivers, converging toward the middle business logic layer. Strategy:

▶ Show Answer
✓ 1 → Top-Down   2 → Bottom-Up   3 → Big Bang   4 → Sandwich
Top-Down (stubs): Start from the highest-level module and work downward. Stubs simulate lower modules. Advantage: user-visible flows can be demonstrated early. Risk: lower-level bugs found late.
Bottom-Up (drivers): Start from utility/foundation modules and work upward. Drivers simulate the calling modules. Advantage: foundational code is solid before higher layers depend on it. Risk: no end-to-end visibility until late.
Big Bang: Integrate everything at once. Simple to set up but very hard to diagnose when something fails — "which module broke it?" Suitable only for very small systems.
Sandwich (Hybrid): Combines Top-Down and Bottom-Up in parallel. Two teams can work simultaneously, meeting in the middle. Reduces overall time but requires careful coordination.
