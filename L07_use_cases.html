Where Use Cases Fit in the Methodology

Deriving use cases is a planning activity that sits in a precise position within the development pipeline:

  1. Acquire preliminary requirements.
  2. Derive use cases ← you are here.
  3. Allocate subsystems to development iterations.
  4. Begin domain modelling.
Purpose of Use Cases

Use cases are derived directly from requirements to ensure the system satisfies the customer's business needs and to help plan the development and deployment schedule.

Core Concepts: Use Cases & Actors

What is a Use Case?

A use case represents a complete business process. To be valid, it must satisfy all four of the following criteria:

#Criterion
1It must be initiated by an actor.
2It must accomplish a useful business task for that actor.
3It must end with the actor explicitly or implicitly acknowledging the task is done.
4It must be a true business process — not a step, sub-task, or system routine.

What is an Actor?

An actor denotes a specific business role played by entities or stakeholders. Key properties:

  • Actors interact with the system and initiate use cases.
  • Actors are strictly external to the system — they are never part of the system itself.
  • Actors are often humans, but can also be hardware, another system, or an external component.
Primary vs. Secondary Actors
  • Primary Actors initiate the use case to achieve their own goal (e.g., "Customer places order").
  • Secondary Actors provide a service to the system to help complete the goal (e.g., a payment gateway API).

What Is NOT a Use Case?

Routine system operations or individual steps are not use cases. The following are common invalid examples:

Invalid Use Cases (Exam Trap)
  • Checking authorization
  • Entering a password
  • Processing data
  • Opening a file
  • Clicking a menu item
  • Traversing a linked list
  • Starting a system

These are all internal steps or system operations, not complete business processes initiated by an actor.

UML Use Case Diagram Notation

ElementVisual RepresentationNotes
Use Case Oval containing the use case name Placed inside the system boundary.
Actor Stick figure Always placed outside the system boundary — even malicious actors like "Hacker".
System Boundary Rectangle enclosing use cases System name at the top. Only use cases and their relationships belong inside.
Association Solid line Connects an actor to a use case. Actor–use case relationships are always associations.
System Boundary Rule

Only use cases (and relationships between them) belong inside the system boundary box. All actors must remain outside, without exception.

Advanced Relationships

To prevent diagrams from becoming overly complex, three advanced relationship types are used:

Inheritance (Generalization)

Indicates that one use case or actor is a specialized version of another.

  • Example: An Admin IS-A Patron. Instead of drawing separate lines from Admin to every use case Patron has access to, you draw the inheritance arrow and inherit all associations.
  • Notation: Solid line with a hollow arrowhead pointing to the generalized (parent) element.

«include» (Inclusion)

Indicates that one use case requires another as a mandatory sub-process.

  • Example: "Checkout" «include»s "Process Payment" — you cannot check out without processing payment.
  • Notation: Dashed arrow pointing to the included use case.

«extend» (Extension)

Indicates that one use case can optionally continue the process of another.

  • Example: "Checkout" can be extended by "Apply Coupon" — coupon application is optional.
  • Notation: Dashed arrow pointing from the extension use case to the base use case.
include vs. extend — Quick Rule
  • «include» → mandatory step, arrow points to what is included.
  • «extend» → optional behaviour, arrow points to the base use case being extended.

Guidelines & Best Practices

  • Avoid overcrowding: Do not show too many use cases in one diagram.
  • Avoid fragmentation: Do not create a separate diagram for every single use case.
  • Avoid overusing extend: Extension should model genuine optional behaviour, not every possible variation.
  • Separation of Concerns / Divide & Conquer: Group related use cases into separate sub-system diagrams (e.g., separate SAMS/User Mgmt from SAMS/Program Mgmt).

Scope, Allocation & Traceability

Defining Use Case Scope

High-level use cases are given precise boundaries using two markers:

  • TUCBW — The Use Case Begins When…
  • TUCEW — The Use Case Ends When…

Allocation Matrix (Project Planning)

A scheduling matrix maps which use cases will be developed during which iteration (Week 1, Week 2, etc.). This lets the team know what needs to be built, what is ready for integration and testing, and tracks overall project status.

Traceability Matrix

A table mapping functional requirements to specific use cases, ensuring nothing is missed.

DirectionPurpose
Forward traceabilityEvery requirement has been built (no requirement is missing from the implementation).
Backward traceabilityNo unnecessary features were built that were not requested.

ClickUP Class Exercise

This is an explicit exam-prep exercise from the lecture on traceability mapping.

Requirements

  • FR1: A Student or TA can view announcements posted by a Lecturer.
  • FR2: An Administrator can delete announcements.

Use Cases

  • UC1: View Announcements
  • UC2: Post Announcements
  • UC3: Delete Announcements

Traceability Mapping

RequirementTraced Use CasesReasoning
FR1 UC1, UC2 Viewing (UC1) and posting (UC2) are both necessary to satisfy FR1 — you cannot view without someone posting.
FR2 UC3 Deleting announcements (UC3) directly satisfies FR2.
Key Takeaway

A single functional requirement can trace to multiple use cases, and a single use case can be traced back to multiple requirements. The matrix makes these many-to-many relationships explicit and auditable.
