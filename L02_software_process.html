Core Concepts of Software Engineering

Software engineering has three primary objectives and operates through three interacting life cycle tracks.

Primary Objectives
  • Significantly improve software productivity and quality.
  • Reduce costs and time to market.

Three Main Tracks

TrackPurpose
Software Development ProcessThe activities that transform requirements into a working system.
Software Quality Assurance (SQA)Ensures activities are performed correctly and artefacts meet requirements.
Software Project ManagementAdministers and controls both development and SQA.

Object-Oriented Software Engineering (OOSE)

A specialisation of SE that views the world and software systems as a collection of objects interacting with one another. This perspective underpins UML, class diagrams, and use case modelling covered in later lectures.

The Software Process & Waterfall Model

What Is a Software Process?

A software process is a series of activities performed to construct a system. It transforms an initial concept into an operational system by identifying business needs, conducting feasibility studies, and formulating requirements.

Each activity produces artefacts (deliverables that become inputs for the next phase), and every phase has strict entrance and exit criteria.

The Waterfall Model

The original, traditional sequential model:

Phase
System Engineering
Requirements Analysis
Software Design
Coding & Unit Testing
Integration Testing
Acceptance Testing
Maintenance
Merits of the Waterfall Model

Its straightforward sequence simplifies project management and naturally supports function-oriented project organisations where specialised teams (e.g., a dedicated design team or testing team) handle each phase sequentially. Documentation is thorough and each phase has a clear end point.

Limitations

Requirements rarely stay fixed. The model is inflexible to change — going back to an earlier phase is costly. This makes it poorly suited to most modern application software development.

Tame vs. Wicked Problems

Not all problems are the same. SE must grapple with a fundamental distinction:

PropertyTameWicked
FormulationCompletely specifiableNo definite formulation
Stopping ruleClear — objectively correct/incorrectNone — solutions are "good" or "bad"
Specification vs. solutionCan be separatedCannot be separated
Cause & effectDefinite chainMulti-causal, unclear
ExamplesMaths, chess, compilers, AI problemsEconomic reforms, application software
Key Insight

Application software development is a wicked problem. You cannot fully specify it before starting, the requirements evolve as stakeholders see the product, and there is no objectively "correct" solution. This is why rigid waterfall often fails and why iterative/agile approaches exist.

Software Process Models

Prototyping Model

Prototypes are built specifically to acquire and validate requirements and assess project feasibility. Once requirements are confirmed, the prototype is typically discarded.

Evolutionary Prototyping

Instead of throwing prototypes away, the prototype directly evolves into the final production system. This avoids wasting time and effort.

  • Suited for: Exploratory systems (e.g., research software) where requirements are genuinely unknown.
  • Not suitable for: Projects requiring a predictable schedule — the end-state is hard to plan upfront.

Spiral Model

A risk-driven approach that loops through phases, choosing its next action based on current risk level:

  • If risks remain → use prototyping.
  • If risks are resolved → proceed like a Waterfall.
  • If the prototype is robust → shift to an Evolutionary model.

Each loop of the spiral represents one iteration of the development cycle, with increasing detail and decreasing risk.

Rational Unified Process (RUP)

An iterative framework with four main phases:

PhaseIterationsKey Deliverables
Inception1–2Simplified use case model, tentative architecture, project plan
ElaborationNArchitectural design, implementation of most critical use cases
ConstructionMultipleRemaining use cases iteratively implemented and integrated
Transition—System deployment, user training, defect correction

PSP & TSP

Personal Software Process (PSP)

A framework of scripts, forms, and standards for training individual engineers and preparing them for team projects. It focuses on disciplined personal habits before scaling to teams.

LevelFocus
PSP0 & PSP0.1Process discipline, baseline measurement, time and defect recording
PSP1 & PSP1.1Size estimation, planning, and scheduling
PSP2 & PSP2.1Quality management — design and code reviews
PSP3.0Component-level development

Team Software Process (TSP)

Scales PSP to a team level. Teams work through cycles of: Strategy → Plan → Requirements → Design → Implementation → Test → Postmortem. The postmortem feeds lessons back into the next cycle.

Quality Assurance & Project Management

Software Quality Assurance (SQA)

SQA operates through two complementary activities:

  • Verification: Ensures development activities are performed correctly — "Are we building the product right?"
  • Validation: Ensures produced artefacts meet requirements — "Are we building the right product?" This includes static checking and dynamic testing.

Software Project Management

Oversees the administration and control of both the development activities and SQA. It tracks schedule, budget, risk, and team coordination to keep the project on track.

CS vs. Software Engineering

DimensionComputer Science (CS)Software Engineering (SE)
GoalPursue optimal solutions"Good enough is enough"
Problem typeTame problemsWicked problems
Scale"Programming in the small""Programming in the large"
FocusTechnical and theoretical issuesAll aspects of the system
CostNot an important considerationHighly important — the Iron Triangle
The Iron Triangle

In SE, project constraints are often framed as PQCT — Productivity/Scope, Quality, Cost, and Time. You can optimise for three, but the fourth will suffer. This trade-off is fundamental to project management in SE.

SE builds upon the foundations of CS and other disciplines — it does not replace them. A software engineer must understand algorithms, data structures, and system theory while also mastering process, communication, and economics.
