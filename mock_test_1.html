Q1 Single MCQ 2 marks

You are the lead engineer for a banking mobile app releasing a new transaction flow. Your constraints: zero downtime is mandatory, rollback must complete in under 2 minutes if error rates spike, and the monitoring team wants real-user performance data before committing to a full rollout. Synthetic CI tests have already passed.

Which deployment strategy best satisfies all three constraints?

▶ Show Answer
✓ Correct Answer: B — Canary
✓ B is correct: Canary routes a configurable percentage of real traffic to the new version. Real-user metrics are collected before full rollout. Rollback is instant — redirect that traffic slice back to the old version.
✗ A (Blue-Green) is wrong: Blue-Green does achieve zero-downtime but it switches ALL traffic at once. It does not collect gradual real-user performance data before full exposure.
✗ C (Rolling) is wrong: Rolling replaces instances gradually but gives no fine-grained traffic control. You cannot limit exposure to 1% of users — eventually all instances are updated.
✗ D (Big Bang) is wrong: Big Bang requires downtime and has no gradual detection mechanism at all.
Canary is named after the canary-in-a-coal-mine practice. It uniquely satisfies both "real traffic data" and "easy rollback" because the traffic split is controlled by a router/load balancer percentage, not by which code version is deployed to which server.

Q2 Multi-Select MCQ 3 marks

Your team has a protected-branch policy: "Never rewrite history on any branch that has been pushed to the shared remote."

Select ALL commands that would violate this policy if run on a branch already pushed to origin. (Select all that apply.)

⚠ Negative Marking — each incorrect selection deducts marks.
▶ Show Answer
✓ Correct: A, B, E, H
✓ A: --amend replaces the last commit with a new SHA. Force-pushing overwrites the remote — history is rewritten.
✓ B: rebase replays commits onto a new base, assigning new SHAs. --force-with-lease still overwrites remote history (it just checks nobody else pushed first).
✓ E: reset --hard HEAD~2 removes two commits from the local branch. Force-pushing makes the remote match — those commits are gone from shared history.
✓ H: Interactive rebase rewrites commit SHAs (squash, reword, drop). Force-pushing propagates those rewrites.
✗ C (revert): Creates a new commit that undoes changes. Existing commits are untouched — safe on shared branches.
✗ D (merge): Adds a merge commit. Does not remove or change existing commits.
✗ F (cherry-pick): Copies a commit as a new commit. Does not alter history.
✗ G (stash): Temporarily shelves working directory changes. No commits are created or modified.
A command "rewrites history" when it changes existing commit SHAs or removes commits. The danger: collaborators' local branches diverge from the rewritten remote, causing messy force-pull situations. revert is always the safe alternative on shared branches.

Q3 Jumbled Sentence 4 marks (1 per blank)

For each system description below, select the architectural style that best fits. Each style may only be used once.

1. The UI layer calls the business-logic layer, which calls the data-access layer. Each layer only knows about the layer directly below it, and changes in the database layer do not propagate upward. →

2. The system is split into an Order Service, Inventory Service, and Payment Service. Each has its own database, is deployed independently, and communicates over lightweight REST APIs. →

3. Raw sensor readings enter a pipeline. Filter A normalises units, Filter B removes outliers, Filter C aggregates by region. Each filter passes its output to the next with no shared state. →

4. When a user uploads a file, no component directly calls another. Instead, an "upload complete" event is published; a transcription service and a thumbnail service each subscribe and act independently. →

▶ Show Answer
✓ 1 → Layered   2 → Microservices   3 → Pipe-and-Filter   4 → Event-Driven
Layered: Strict dependency hierarchy — each layer only uses the layer below. Protects upper layers from lower-layer changes.
Microservices: Independently deployable services, each owning its own data store, communicating via APIs. Enables team autonomy and independent scaling.
Pipe-and-Filter: Sequential transformation pipeline. Each filter is stateless and reusable. Excellent for data processing workflows.
Event-Driven: Publishers emit events without knowing who handles them. Subscribers react independently. Decouples components temporally and spatially.
The key discriminator between Pipe-and-Filter and Event-Driven: P&F is sequential and the output of one filter flows directly into the next; Event-Driven has no direct flow — any number of subscribers react to an event broadcast.

Q4 Matching 4 marks (1 per match)

Match each software process model description to the correct model name.

⚠ Negative Marking — each incorrect match deducts marks.
A linear, sequential model where each phase (Requirements → Design → Implementation → Testing → Maintenance) must be fully completed before the next begins.
Structures development into four phases: Inception, Elaboration, Construction, and Transition — with multiple iterations inside each phase. Use-case driven and architecture-centric.
Each cycle involves risk analysis before proceeding. Represented as a spiral where each loop addresses a new development objective and risk level determines how far to proceed.
A discipline framework targeting individual developers — focuses on personal planning, time tracking, size and defect measurement, and quality goals for one engineer's work.
▶ Show Answer
✓ Row 1 → Waterfall   Row 2 → RUP   Row 3 → Spiral   Row 4 → PSP
Waterfall: Sequential phases. Simple to manage but inflexible — you cannot go back once a phase ends. Best for well-understood, stable requirements.
RUP (Rational Unified Process): Iterative and use-case driven. The four phases are not equal — most development happens in Construction, but architecture is locked down during Elaboration.
Spiral: Risk-driven. Each spiral loop asks: what are the objectives? what are the risks? prototype/build. plan next iteration. Suitable for large, high-risk projects.
PSP (Personal Software Process): Watts Humphrey's model for individual engineers. Scales up to TSP (Team Software Process) for team-level discipline.
Common trap: confusing RUP with Agile. RUP is iterative but not Agile — it is heavyweight, document-driven, and prescribes specific roles and artefacts. Agile (Scrum, XP) is lightweight and adaptive.

Q6 Fill in the Blank 2 marks (0.5 per blank)

Consider a domain model with the following entities: Customer, Order, OrderItem, and Product. Fill in the blanks based on your understanding of UML domain modelling.

Blank 1: A Customer can place Orders (use UML notation or describe in words).
Blank 2: The OrderItem entity, which connects Order and Product to record quantity and price per line, is an example of an class.
Blank 3: An OrderItem cannot exist without a parent Order. This is a relationship.
Blank 4: A Library contains Books, but the Books could exist independently in another library. This is an relationship.
▶ Show Answer
✓ Blank 1: 1..* (one or many)  |  Blank 2: associative  |  Blank 3: composition  |  Blank 4: aggregation
Blank 1 (1..*): A customer must place at least one order to be meaningful in this system (or zero if they registered but haven't ordered — could also be 0..*). The key is understanding multiplicity notation.
Blank 2 (associative class): OrderItem exists purely because of the relationship between Order and Product — it holds attributes of that relationship (quantity, price at time of purchase). This is the classic pattern for resolving many-to-many relationships with additional attributes.
Blank 3 (composition): In composition (filled diamond), the child's lifecycle depends entirely on the parent. If the Order is deleted, all its OrderItems are deleted too.
Blank 4 (aggregation): In aggregation (hollow diamond), the part can exist without the whole. Books outlive any one library's collection.
The composition vs. aggregation distinction is frequently tested. Remember: composition = cannot exist without parent (filled diamond), aggregation = can exist independently (hollow diamond).

Q7 Multi-Select MCQ 4 marks

A student team is modelling a university portal. They have brainstormed the following items for their use case diagram.

Select ALL items that qualify as valid use cases. (Select all that apply.)

⚠ Negative Marking — each incorrect selection deducts marks.
▶ Show Answer
✓ Correct: A, C, E, G, I
✓ A, C, E, G, I are valid use cases — they are user-observable goals: things an actor wants to accomplish with the system. Each can be described as "actor does X to achieve Y outcome."
✗ B (regex validation): An internal implementation detail. The actor's goal is "submit valid data" — the validation mechanism is hidden inside the system.
✗ D (insert DB record): A database operation triggered internally. Actors do not interact with the database directly.
✗ F (hash password): A security implementation step, not a user-visible goal. Actors never "choose to hash" — they log in.
✗ H (click button): A UI interaction, not a goal. Use cases abstract above UI mechanics.
✗ J (session management): Internal system infrastructure, invisible to actors.
The test for a valid use case: can you phrase it as "Actor verb-phrase in order to achieve goal"? If the sentence only makes sense from inside the system's implementation, it is not a use case.

Q8 Matching 5 marks (1 per match)

Match each integration testing strategy to the description that best characterises it.

⚠ Negative Marking — each incorrect match deducts marks.
All modules are developed and tested in isolation, then integrated and tested together all at once at the end.
Integration begins from the highest-level control modules. Lower-level modules that are not yet available are replaced by stubs.
Integration begins with the lowest-level utility modules. Higher-level modules that call them are replaced by drivers until they are ready.
Combines top-down and bottom-up simultaneously: top layers are tested with stubs while bottom layers are tested with drivers; the two fronts meet in the middle.
Each new component is integrated and tested as soon as it is ready, without waiting for all modules to be complete. Defects are found close to when the component was built.
▶ Show Answer
✓ Row 1 → Big Bang   Row 2 → Top-Down   Row 3 → Bottom-Up   Row 4 → Sandwich   Row 5 → Incremental
Big Bang: Simple but risky — when integration fails you don't know which module caused it. Fault isolation is very hard.
Top-Down + Stubs: Can demonstrate the high-level flow early. Stubs simulate lower modules with hardcoded responses. Real lower modules replace stubs progressively.
Bottom-Up + Drivers: Solid foundation testing. Drivers simulate the calling module. No end-to-end flow visible until near the end.
Sandwich: Hybrid that captures benefits of both. Complex to coordinate but reduces overall integration risk for large systems.
Incremental: Best aligned with Agile practices — continuous integration adds new modules to a running test suite continuously.
Key memory hook — Top-Down uses stubs (stubs go below, standing in for not-yet-built lower modules); Bottom-Up uses drivers (drivers go above, calling the lower modules being tested).

Q9 Single MCQ 2 marks

A Scrum team's Daily Scrum consistently overruns by 25 minutes. Developers use it to give detailed implementation updates, discuss architectural decisions, and report to the Scrum Master. The Product Owner attends to re-prioritise the backlog mid-meeting.

What is the fundamental purpose of the Daily Scrum that is being violated?

▶ Show Answer
✓ Correct Answer: B
✓ B is correct: The Daily Scrum is owned by the Developers (not the Scrum Master). It is time-boxed to 15 minutes. The Developers inspect their sprint progress and adapt their daily plan — it is an inspect-and-adapt event, not a reporting ceremony.
✗ A: The Daily Scrum is not a status report to anyone. Reporting to the Scrum Master turns it into a hierarchical check-in, which undermines team self-management.
✗ C: Backlog prioritisation is the Product Owner's responsibility and happens in Backlog Refinement or Sprint Planning — not the Daily Scrum.
✗ D: Technical deep-dives should be separate meetings ("after-parties") attended only by relevant people, not the whole team for 15 minutes.
A common CT trap: the question may say "raises impediments with the Scrum Master" — this sounds plausible but is wrong. Impediments may surface during the Daily Scrum, but the Scrum Master addresses them outside it. The core purpose is the Developers' own inspection-adaptation loop.

Q10 Jumbled Sentence 3 marks (1 per blank)

A test engineer is categorising test activities for a healthcare platform. Fill in each blank with the correct non-functional testing type from the word bank.

Word bank: load, stress, compatibility, availability, security, usability, performance

1. The team gradually increases the number of concurrent users from 100 to 10,000 while measuring response times and throughput to find the system's degradation threshold. This is testing.

2. The team pushes the system to 300% of its expected peak traffic to observe whether it crashes gracefully or corrupts data, and how it recovers when load is removed. This is testing.

3. The team verifies that the patient portal works correctly on Chrome, Firefox, Safari, Android 13, iOS 17, desktop, and tablet screen sizes. This is testing.

▶ Show Answer
✓ 1 → load   2 → stress   3 → compatibility
Load testing: Operates within expected capacity — the goal is to measure performance characteristics at realistic and peak expected usage levels.
Stress testing: Operates beyond expected capacity — the goal is to find the breaking point and observe failure modes. Questions answered: does the system crash or degrade gracefully? Does it recover?
Compatibility testing: Verifies consistent behaviour across different environments — operating systems, browsers, devices, screen resolutions, and sometimes network conditions.
Load vs. Stress: load = "how does it perform?" (still within bounds); stress = "how does it fail?" (pushed beyond bounds). The exam question often uses phrases like "until degradation" for load and "beyond capacity" or "observe crashes" for stress.
