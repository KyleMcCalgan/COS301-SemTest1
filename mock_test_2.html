Q1 Jumbled Sentence 4 marks (1 per blank)

A software architect is choosing an architectural style for each new project below. Select the best-fit style from the word bank.

Word bank: Layered, Microservices, Pipe-and-Filter, Client-Server, Event-Driven

1. A desktop accounting application connects to a central database server. Multiple workstations request data and the server manages all data access and business rules. →

2. A continuous integration system reads source code, runs a linter, then a compiler, then unit tests, then a code-coverage analyser — each step consumes the previous step's output. →

3. A large fintech platform has a Payments team, a Fraud Detection team, and a Notifications team — each team owns its own service, deploys independently, and calls others only via versioned APIs. →

4. A smart home system: when a motion sensor fires, a security camera starts recording, a mobile notification is pushed, and exterior lights turn on — none of these components know about each other. →

▶ Show Answer
✓ 1 → Client-Server   2 → Pipe-and-Filter   3 → Microservices   4 → Event-Driven
Client-Server: Clients make requests; one central server manages shared resources and responds. The distinction from Microservices is that there is a single controlling server, not independent peer services.
Pipe-and-Filter: Sequential processing pipeline — output of step N is the input to step N+1. Each filter is independent and stateless. Classic for data transformation workflows like CI pipelines and ETL.
Microservices: Independently deployable units owning their own data. Teams are autonomous. Key markers: own database per service, communicate via API.
Event-Driven: Publishers emit events; subscribers react with no knowledge of each other. Key marker: "none of these components know about each other" — decoupled via an event bus/broker.
Distinguishing Client-Server from Microservices: Client-Server has a central, authoritative server that all clients depend on. Microservices have many peer services — each can be both a "client" and a "server" to others.

Q2 Single MCQ 1 mark

In Git's internal object model, what exactly is a commit?

▶ Show Answer
✓ Correct Answer: C
✓ C: Git stores the full state of every tracked file as a tree of objects. Unchanged files are referenced by their hash (deduplication), not re-stored. The commit object also points to its parent(s), forming the DAG (Directed Acyclic Graph).
✗ A (delta/diff): This is how many older VCS (SVN, CVS) work. Git is snapshot-based, not delta-based. This is why Git operations like branching and log are fast — you don't need to reconstruct state by replaying diffs.
✗ B: Only changed files would still be a partial picture. Git stores the complete tree (though it re-uses blob objects for unchanged files).
✗ D: A commit is a local object. Pushing sends it to origin, but the commit itself exists locally first.
The snapshot model is foundational to understanding why git checkout, git reset, and branching are cheap operations — Git just moves a pointer to a different snapshot node in the DAG.

Q3 Matching 4 marks (1 per match)

Match each testing tool to its primary purpose.

⚠ Negative Marking — each incorrect match deducts marks.
JMeter
OWASP ZAP
Google Lighthouse
SonarQube
▶ Show Answer
✓ JMeter → Load/performance   OWASP ZAP → Security   Lighthouse → Performance/accessibility/SEO   SonarQube → Static analysis
JMeter (Apache): Open-source load testing tool. Simulates concurrent users sending HTTP requests. Measures throughput, response time, and error rates. Used for load and stress testing.
OWASP ZAP (Zed Attack Proxy): Free security testing tool that intercepts and analyses web traffic, actively scanning for OWASP Top-10 vulnerabilities like SQL injection and XSS.
Google Lighthouse: Automated auditing tool built into Chrome DevTools. Scores web pages on performance (Core Web Vitals), accessibility (WCAG), best practices, and SEO.
SonarQube: Runs static analysis on source code without executing it. Reports code smells, bugs, vulnerabilities, and tracks technical debt over time. Integrates with CI pipelines.
Common exam trap: confusing JMeter with SonarQube. JMeter tests the running application under load (dynamic, black-box). SonarQube analyses the source code without running it (static, white-box).

Q4 Multi-Select MCQ 3 marks

A Scrum team is beginning Sprint Planning. The Scrum Master reminds the team of the two key questions Sprint Planning must answer.

Select the TWO questions that Sprint Planning is designed to answer. (Negative marking applies.)

⚠ Negative Marking — each incorrect selection deducts marks.
▶ Show Answer
✓ Correct: B and D
✓ B — "What can be done?": The team and Product Owner select items from the top of the Product Backlog that can be completed within the sprint timebox. This creates the Sprint Backlog.
✓ D — "How will it be done?": The Developers plan the technical work — breaking stories into tasks, identifying dependencies, and creating a delivery approach. This is owned by the Developers, not the Product Owner.
✗ A: Individual task assignment is not mandated by Scrum. Teams are self-organising — who does what emerges during the sprint.
✗ C: Sprint length is set once (usually 2–4 weeks) and is consistent across sprints. It is not decided during each Sprint Planning.

Sprint Planning has two distinct parts: (1) "What" — the Product Owner presents priorities and the team selects items, forming the Sprint Goal; (2) "How" — Developers decompose selected items into tasks and estimate effort.

A domain model shows: Doctor and Patient have a many-to-many relationship. An Appointment entity sits between them, recording date, time, and diagnosis notes. A Hospital has many Doctors; if the Hospital closes, the Doctors' employment records are deleted too.

Q6 Matching 4 marks (1 per match)

Match each Agile/Scrum governance concept to its correct definition.

⚠ Negative Marking — each incorrect match deducts marks.
Velocity
Burndown Chart
Definition of Ready
Sprint Goal
▶ Show Answer
✓ Velocity → story points/sprint   Burndown → remaining work chart   DoR → entry criteria   Sprint Goal → sprint objective
Velocity: Measured in story points completed per sprint. After several sprints it stabilises and can be used to forecast release dates ("at this velocity we'll complete the backlog in ~5 more sprints").
Burndown Chart: X-axis = sprint days, Y-axis = remaining story points. Ideal trend is a diagonal line to zero. A flat or rising line signals the team is behind.
Definition of Ready (DoR): A backlog item is "ready" when: the acceptance criteria are written, it is estimated, dependencies are identified, and the team understands it well enough to start. DoR prevents half-baked stories from entering sprints.
Sprint Goal: A single sentence agreed during Sprint Planning that describes what value the sprint will deliver. It gives the team flexibility: if one story blocks, they can pivot to another story that still serves the same goal.
Don't confuse Definition of Ready (DoR) with Definition of Done (DoD). DoR = entry criteria for pulling a story into a sprint. DoD = exit criteria for declaring a story complete.

Q7 Single MCQ 2 marks

A platform team is deploying a new API gateway. Requirements: (a) if the new version causes errors, the team must be able to roll back in under 30 seconds; (b) both the old and new versions must be able to run simultaneously so A/B testing is possible; (c) downtime is unacceptable.

Which deployment strategy satisfies all three requirements?

▶ Show Answer
✓ Correct Answer: A — Blue-Green
✓ A (Blue-Green): Both environments run simultaneously. Rollback = flip the load balancer back to Blue, which takes seconds. Zero downtime because traffic switches at the LB level, not at the instance level.
✗ B (Canary): Canary also satisfies zero-downtime and simultaneous versions, BUT rollback requires re-routing traffic percentage — it is not typically sub-30-second for full rollback. Also, the scenario doesn't mention gradual real-traffic testing, which is Canary's distinguishing advantage.
✗ C (Rolling): Rolling updates instances one by one. During the update window, both versions serve traffic, but rollback requires re-rolling all instances back — not instantaneous.
✗ D (Feature Flag): A code-level toggle, not a deployment strategy. Does not maintain two separate environments.
Blue-Green's key advantages: instant rollback and clean environment separation. Its cost: maintaining double the infrastructure at all times. Canary is better when you want gradual real-user validation; Blue-Green is better when you want instant switchover.

Q8 Multi-Select MCQ 4 marks

A developer writes an integration test for OrderService. The test calls the real PaymentService, the real InventoryService, and the real CartService — no mocks are used. The test asserts only that a successful order returns HTTP 200.

Select ALL statements that correctly identify issues with this test. (Select all that apply.)

⚠ Negative Marking — each incorrect selection deducts marks.
▶ Show Answer
✓ Correct: B and D
✓ B — Non-determinism: Real services introduce unpredictable outcomes: network timeouts, database state from other tests, external payment gateway rate limits. The test can pass today and fail tomorrow with no code change.
✓ D — No intermediate validation: The test only checks the happy path (HTTP 200). It does not verify: inventory rollback on payment failure, partial-order states, idempotency, or error message format. These are critical integration concerns.
✗ A: The opposite is true — real services mean the environment is NOT controlled or predictable.
✗ C: Integration tests CAN use real services in a staging environment. The issue is not which environment — it's that the test is non-deterministic and incomplete.
✗ E: The test calls three real services — it is absolutely an integration test, not a unit test. Unit tests isolate a single unit with mocks for all dependencies.
✗ F: Only the success path is tested. Full integration testing requires error, boundary, and failure scenario coverage.
Integration tests should test the contract between services, including failure modes. Mocks/stubs are typically used to control the environment and test specific scenarios (e.g., "payment gateway returns 402") — not to avoid real code paths.

Q9 Jumbled Sentence 4 marks (1 per blank)

Fill in the blanks to correctly describe the three areas of Git and how files flow between them.

1. When you edit a file in your project directory, the change exists in the and Git considers it "modified" but not yet tracked for the next commit.

2. Running git add <file> moves the snapshot of that file into the (also called the index), marking it as part of the next commit.

3. Running git commit saves the staged snapshot permanently into the as a new commit object in the DAG.

4. Running git push uploads the new commits from the local repository to the (e.g., GitHub/origin).

▶ Show Answer
✓ 1 → working directory   2 → staging area   3 → local repository   4 → remote repository
Working Directory: Your actual files on disk. Modified files here are tracked by Git but not yet queued for commit. git status shows them as "Changes not staged".
Staging Area (Index): A preparation zone. git add snapshots the file's current state into the index. You can stage some files but not others, giving fine-grained control over commits.
Local Repository (.git folder): The permanent history store. Each git commit creates a new immutable commit object pointing to a tree of blobs.
Remote Repository (origin): A shared server copy. git push sends local commits to origin; git pull/fetch retrieves remote commits.
The staging area is Git's unique advantage over simpler VCS tools. It lets you craft precise, atomic commits — stage only the changes relevant to one logical unit of work, even if multiple files changed at the same time.

Q10 Single MCQ 2 marks

In Object-Oriented Software Engineering (OOSE), how is the world fundamentally viewed?

▶ Show Answer
✓ Correct Answer: B
✓ B: OOSE views the world as objects — entities that encapsulate state (attributes) and behaviour (methods) — interacting via messages/method calls to produce system behaviour. This is the foundational paradigm of OO design.
✗ A: This describes procedural/structured programming — the paradigm OOSE was designed to improve upon by moving from "procedures acting on data" to "objects responsible for their own data and behaviour."
✗ C: This describes functional programming — no mutable state, pure functions. The opposite of the stateful, message-passing OO model.
✗ D: A relational model — a data storage paradigm, not a software design worldview.
OOSE (Ivar Jacobson) introduced use cases as the mechanism for capturing how objects collaborate to deliver system behaviour. The key insight: objects are first-class citizens with identity, state, and responsibility — not passive data containers.
