Q1 Single MCQ 2 marks

A new developer joins the team and asks: "What is the first Scrum ceremony of a sprint — the one that officially starts it?"

Which ceremony defines the start of a sprint?

▶ Show Answer
✓ Correct Answer: B — Sprint Planning
✓ B: Sprint Planning is the first event of the sprint. It creates the Sprint Backlog (selected items + decomposed tasks) and establishes the Sprint Goal. The sprint clock officially starts when Sprint Planning begins.
✗ A (Daily Scrum): The Daily Scrum happens throughout the sprint, starting from Day 1. It does not start the sprint — the sprint has already started by Sprint Planning.
✗ C (Backlog Refinement): Refinement is a recommended practice but is NOT an official Scrum event. It happens mid-sprint in preparation for the next Sprint Planning — not at the start of the current sprint.
✗ D (Sprint Review): Sprint Review happens at the END of the sprint, not the start.
Sprint order: Sprint Planning → (Daily Scrums throughout) → Sprint Review → Sprint Retrospective → (next) Sprint Planning. Sprint Planning = start. Review + Retro = end.

Q3 Jumbled Sentence 4 marks (1 per blank)

A QA team is categorising test activities for an online banking platform. Fill in each blank with the correct non-functional testing type.

Word bank: load, stress, security, compatibility, availability, usability

1. The team simulates 50,000 simultaneous users making transfers and measures response times, throughput, and error rates at expected peak load. This is testing.

2. The team uses OWASP ZAP to scan the login API for SQL injection and cross-site scripting vulnerabilities, and performs penetration testing on the authentication flow. This is testing.

3. The team verifies the bank's SLA of 99.95% uptime by monitoring whether planned and unplanned outages breach the annual downtime budget. This is testing.

4. The team asks 10 real customers to perform 5 common banking tasks while a UX researcher observes, measuring task completion rates, error rates, and time-on-task. This is testing.

▶ Show Answer
✓ 1 → load   2 → security   3 → availability   4 → usability
Load testing: Performance under expected (high but realistic) user volumes. Identifies bottlenecks before they affect users.
Security testing: Identifies vulnerabilities that could allow unauthorised access, data breaches, or system compromise. Tools: OWASP ZAP, Burp Suite. Techniques: penetration testing, static analysis, code review.
Availability testing: Verifies the system meets uptime SLA targets. Measures MTTF (Mean Time to Failure) and MTTR (Mean Time to Recovery).
Usability testing: Measures how effectively real users can accomplish tasks. Not automated — requires human participants. Outputs: task success rate, error frequency, user satisfaction scores.
Security testing vs. penetration testing: "pentesting" is a subset of security testing — specifically the active attack simulation component. Broader security testing includes static analysis, threat modelling, and compliance checks. The exam accepted "pentesting" as partially correct for blank 3 — but "security" is the broader and more accurate category name.

Q4 Matching 4 marks (1 per match)

Match each integration testing description to the correct test double type or strategy keyword.

⚠ Negative Marking — each incorrect match deducts marks.
A simplified replacement that simulates a lower-level module not yet built — it returns hardcoded responses when called by the module under test.
A replacement that simulates a higher-level calling module — it invokes the lower-level module being tested so that it can be tested independently.
A test double that, in addition to simulating a dependency, also records how it was called — verifying that the system under test made the expected calls with the expected parameters.
A test double that wraps a real object and records how it is called, but delegates actual execution to the real implementation — useful when you want to observe real behaviour.
▶ Show Answer
✓ Row 1 → Stub   Row 2 → Driver   Row 3 → Mock   Row 4 → Spy
Stub: Passive — it stands in place of a not-yet-built lower dependency, returning predetermined responses. Used in Top-Down integration. No verification of calls.
Driver: Active — it calls the module being tested, simulating a higher-level caller not yet built. Used in Bottom-Up integration.
Mock: A stub with memory — it verifies that the system under test called it in the right way. Used in unit tests to assert collaboration between components.
Spy: A wrapper around a real object. The real code runs, but calls are recorded. Useful when you want to observe real behaviour but also track interactions.
Quick summary: Stub = fake with no verification. Driver = fake caller. Mock = fake with call verification. Spy = real object with observation. In practice, "mock" is often used loosely to mean any test double — be precise in exam answers.

▶ Show Answer
✓ Blank 1: working directory  | Blank 2: git add  | Blank 3: git stash  | Blank 4: git fetch
Working directory: Your actual project files. git status compares them to the staging area and last commit.
git add: Stages specific files or patterns. git add . stages all tracked changes. Critical: only staged changes are included in the next git commit.
git stash: Saves modified tracked files to a stash stack. Reverts the working directory to the last commit state. git stash pop restores the stashed changes.
git fetch: Downloads remote commits to your local repository (FETCH_HEAD / remote tracking branches) but does NOT merge them. git pull = git fetch + git merge.

Q6 Single MCQ 2 marks

A developer is practising Test-Driven Development. She has just identified a new feature to implement.

What is the FIRST step in the TDD Red-Green-Refactor cycle?

▶ Show Answer
✓ Correct Answer: B — Write a failing test first (Red)
✓ B (Red): The test is written BEFORE any implementation. It should fail because the feature doesn't exist yet. A test that passes before implementation suggests either the test is wrong or the feature already exists.
✗ A: Writing implementation first is traditional development, not TDD. TDD inverts this: tests define the API and expected behaviour first.
✗ C (Refactor): Refactoring is the THIRD step — after you have green (passing) tests. You never refactor without a green test suite to protect you.
✗ D: While running the existing suite before starting is good hygiene, it is not the first step of the TDD cycle itself.
TDD cycle: Red (write failing test) → Green (write minimum code to pass the test) → Refactor (clean up code while keeping tests green) → back to Red for the next requirement. The discipline of writing the test first forces you to think about the API before the implementation.

Q7 Multi-Select MCQ 5 marks

A social media platform has its user profile database sharded by user_id. Each shard has 1 primary write node and 2 async read replicas. A Redis cache with a 10-minute TTL sits in front of profile reads. The system uses no idempotency keys on write operations.

Select ALL statements that are technically correct about this system. (Negative marking applies.)

⚠ Negative Marking — each incorrect selection deducts marks.
▶ Show Answer
✓ Correct: A, C, D, F
✓ A — Replication lag: Async replication does not guarantee immediate visibility. Reads from replicas immediately after a write may see the old value. This is "eventual consistency."
✓ C — Cache staleness compounds replication lag: Even if the replica converges quickly, the cache TTL (10 min) can serve the stale cached value for much longer. Total potential staleness = replication lag + cache TTL.
✓ D — Read/write throughput independence: Read replicas distribute read load. But writes always go to the primary — adding replicas has no effect on write capacity. Write throughput scales with shards, not replicas.
✓ F — No idempotency keys: Client retries after timeout → server may have already processed the first request. Without idempotency, the operation executes twice → duplicates.
✗ B: Sharding by user_id is efficient for single-user lookups, but cross-user analytics (e.g., "find all users in Cape Town") require querying ALL shards in parallel — not horizontally scalable for that pattern.
✗ E: This describes SYNCHRONOUS replication. In async replication, the primary acknowledges the write IMMEDIATELY after writing locally — replicas catch up asynchronously.
✗ G: The shard router / load balancer is a shared component. If it fails, ALL shards are impacted. Shard-level failures can also affect shared infrastructure (monitoring, routing). This statement is false as written.

Q8 Matching 4 marks (1 per match)

Match each software process model to its defining characteristic.

⚠ Negative Marking — each incorrect match deducts marks.
Organises work into fixed-length iterations called sprints; teams self-organise, requirements emerge over time, and working software is the primary measure of progress.
All requirements must be fully defined before design begins; design must be complete before coding; changes to earlier phases are expensive and discouraged.
At the start of each loop, the team determines objectives; analyses risks; selects a strategy based on risk level; builds and tests a prototype; then plans the next loop.
An iterative and incremental framework with four phases (Inception, Elaboration, Construction, Transition) and six disciplines running in parallel. Use-case driven and architecture-centric.
▶ Show Answer
✓ Row 1 → Agile/Scrum   Row 2 → Waterfall   Row 3 → Spiral   Row 4 → RUP
Agile/Scrum: Embraces change, ships working software frequently (every sprint), measures success by working software not documentation.
Waterfall: Sequential and phase-gated. Best for stable, well-understood requirements. The cost of change increases exponentially as the project progresses.
Spiral: Risk-driven iteration. Each loop = plan objectives → analyse risks → build prototype → review. If risk is too high, the project can be terminated early. Suitable for large, high-uncertainty projects.
RUP: Iterative but heavyweight. Unlike Agile, RUP produces many artefacts and has specific roles. The architecture is finalised in the Elaboration phase before most development (Construction) begins.
Key discriminator between RUP and Agile: RUP is still largely plan-driven and produces formal artefacts (documents, diagrams, specifications). Agile prioritises "working software over comprehensive documentation." Both are iterative, but their philosophy differs.

1. When a resident submits a report about a burst water pipe, a record is created in the system with the location, category, and priority of the reported issue.

2. The team reviews the fault record, validates its priority, and allocates available resources to address it.

3. Once resources are allocated, the system creates a that specifies the task, assigned crew, equipment, and estimated time of arrival.

4. After the pipe is repaired, the team updates the work order status to "Resolved" in the system, triggering a notification to the resident.

▶ Show Answer
✓ 1 → Fault   2 → Dispatch   3 → Work Order   4 → FieldOps
Fault: The initial record of the reported problem — it captures what was reported, by whom, where, and at what severity.
Dispatch: The coordination function — evaluates faults, decides priority, assigns resources. This is an actor or organisational role in the use case model.
Work Order: The operational instruction created after dispatch decides to address a fault. It contains all the information a field crew needs to do the job.
FieldOps: The field operations team that physically performs the repair. They update the system when work is complete, closing the loop on the fault report.
This flow (Fault → Dispatch → Work Order → FieldOps) is the core use case of a Municipal Services Management System — the type of system commonly used in the COS301 mini-project. The entities map directly to use cases, actors, and domain model classes.
