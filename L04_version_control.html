Why Version Control?

Version control systems (VCS) are not just a convenience — they are critical infrastructure for any serious software project. Their value becomes most apparent during disasters.

What a VCS Enables During a Production Crisis
  • Identify exactly what code changed between versions.
  • Roll back immediately to a known-good, stable version.
  • Trace accountability — who authored and approved the change?
  • Investigate why testing did not catch the defect before release.

Centralized vs. Distributed Models

The Centralized Model (Before Git)

Older systems (CVS, SVN) relied on one central server that held the "true" version of the code. Developers checked out files, and all history and commits were stored in this single central location.

Limitations of Centralized VCS
  • Single point of failure — if the server goes down, no one can commit.
  • Requires network connectivity to perform most operations.
  • Poor support for branching and merging workflows.

Git — The Distributed Model

In Git, every developer's machine holds a full copy of the repository, including the complete history. Operations like branching, merging, and viewing history are local and instant. Remote servers (like GitHub) are just another repository that teams agree to treat as authoritative.

Git Commits & the DAG

Git as a Directed Acyclic Graph (DAG)

Git operates mathematically as a Directed Acyclic Graph:

  • Nodes represent commits.
  • Directed edges are pointers moving from a child commit to its parent commit(s).
  • A regular commit has one parent.
  • A merge commit has two or more parents.
  • Acyclic — you can never follow edges back to a previous commit; time only moves forward.

What Is a Commit?

A commit is not just a diff of changes — it is a full snapshot of the entire project state at that moment. Every commit contains:

ComponentDescription
Tree pointerPoints to the directory structure and file contents at that moment.
Parent pointer(s)Links to previous commits, forming the DAG.
MetadataAuthor name, email, timestamp, and commit message.
Cryptographic hashA SHA-1/SHA-256 hash that uniquely identifies this exact commit and all its ancestors.
Why Hashes Matter

Because each hash is computed from the commit's content and its parent's hash, changing any commit in history changes all descendant hashes. This makes Git history tamper-evident — you cannot silently alter old commits.

The Three Areas of Git

AreaAlso Known AsRole
Working Directory Working Tree Your local file system — where you actively edit files. Changes here are untracked until staged.
Staging Area Index An intermediate holding area. You git add files here to prepare them for the next commit. Lets you craft precise, atomic commits.
Local Repository .git directory Where Git safely stores all committed snapshots, branches, tags, and metadata. The complete project history lives here.
Flow

Edit in Working Directory → git add → Staged in Index → git commit → Stored in Local Repo → git push → Remote (GitHub/GitLab).

CI/CD Deployment Strategies

Continuous Integration / Continuous Deployment (CI/CD) pipelines automate the process of building, testing, and releasing software. Two key deployment risk-mitigation strategies are:

Canary Deployment

A new version is deployed to a very small percentage of users first (e.g., 5%). The team observes errors, performance metrics, and user behaviour before gradually increasing the rollout to all users.

  • Advantage: Limits blast radius — a bad release only affects a fraction of users.
  • Advantage: Real production traffic validates the new version before full rollout.
  • Disadvantage: Requires infrastructure to route traffic to multiple versions simultaneously.

Rolling Deployment

Old application versions are gradually replaced with new ones across servers, one server (or group) at a time. At no point are all servers taken offline simultaneously.

  • Advantage: Minimal downtime — the service remains available throughout.
  • Advantage: Significantly reduces the risk of a full system failure.
  • Disadvantage: During the rollout, two versions run simultaneously — requires backwards compatibility.

Troubleshooting Scenarios

These three disaster scenarios are classic exam and real-world situations. Know them cold.

Scenario 1 — CI Passed but Production Crashed

Situation

All CI tests passed, but the application crashed immediately after deploying to production.

Root cause: This is almost always caused by an environmental mismatch between the CI environment and production:

  • Production has different database data or schema that CI's test database lacks.
  • Missing or different environment variables (API keys, feature flags, connection strings).
  • Different server configurations, OS versions, or dependency versions.

Fix: Make CI environments as production-like as possible. Use containerisation (Docker) and infrastructure-as-code to eliminate environment drift.

Scenario 2 — Force-Push Erased Shared Branch History

Situation

A teammate rebased and force-pushed to a shared feature branch, rewriting history and causing others to lose their commits.

Fix:

  1. Immediately tell the team to stop pushing/pulling to that branch.
  2. Find a developer whose local machine still has the pre-rebase state, or use git reflog to find the hash of the branch before the rebase.
  3. Force-push the correct history back to the remote server.

Prevention: Protect shared branches from force-pushes in your repository settings. Reserve rebasing for private local branches only.

Scenario 3 — git reset --hard Lost Commits

Situation

A developer ran git reset --hard on their local machine and lost several commits that had not been pushed to the remote.

Fix:

git reflog

The reflog acts as a diary of all local HEAD movements. Find the hash of the commit that existed right before the hard reset, then:

git reset --hard <commit-hash>

This recovers the lost state. Note: git reflog only works for local operations and expires after ~90 days by default.

The Golden Rule of git reflog

Almost nothing is truly lost in Git as long as you act before garbage collection runs. git reflog is your first tool to reach for in any local history disaster.
