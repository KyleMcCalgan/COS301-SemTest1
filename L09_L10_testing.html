What Is Software Testing?

Software testing is a dynamic validation activity — it executes the program to detect defects and demonstrate that software satisfies its requirements and constraints.

Core Principles
  • Testing is dynamic — review and inspection alone are not enough. You can't test a car without driving it.
  • Testing can detect errors but cannot prove a program is error-free.
  • A good test case has a high probability of finding an undetected error.
  • A successful test is one that uncovers an undiscovered error.
  • Testing ≠ Debugging. Debugging locates and fixes a known error; testing finds unknown ones.

Why Software Testing Matters

Testing costs 40–80% of total development costs (higher for real-time embedded systems). The average field-detected bug cost US$15,000 in 1991 — catching it in testing is far cheaper. Testing is especially critical for: instrumentation systems, process control, military systems, and safety-critical healthcare software.

Verification & Validation

ConceptQuestionFocus
Verification"Are we building the product right?"Conformance to specification
Validation"Are we building the right product?"Meeting user needs

Three Approaches to Correctness

  • Testing — pragmatic; proves errors exist but cannot assert a program is error-free.
  • Program correctness proof — theoretical; once proved, correct forever.
  • Symbolic execution — hybrid; executes programs using symbolic values and correctness proof rules.

Test Case Generation Techniques

White Box Testing

Based on knowledge of the internal workings of the product. Test cases are designed from program structure and control flow.

TechniqueWhat It Tests
Basis Path TestingExercise all control flow paths
Condition TestingEvery outcome of every condition
Data Flow TestingDefine/use relationships of data elements

Black Box Testing

Based on the functional specification. No knowledge of internal structure required.

TechniqueWhat It Tests
Boundary Value AnalysisTest cases from boundary values of variables
Causal-Effect AnalysisDerived from stimuli/responses and input-output relationships
Equivalence PartitioningDerived from partitions of input/output domains

Test Categories & The Pyramid

LevelDescription
Unit TestingIndividual module level — functions, classes
Integration TestingInterfaces and interactions between modules
Validation TestingAgainst customer's functional requirements
System TestingRecovery, Security, Stress, Performance

The testing pyramid: Unit Tests (bottom — fastest, most isolated) → Integration → API → UI → Manual (top — slowest). Write most tests at the bottom.

Unit Tests — Properties & AAA

A unit test targets the smallest possible piece of testable software — usually a single function or method — in isolation from databases, APIs, and UIs.

Properties of a Good Unit Test
  • Fast — runs in milliseconds.
  • Isolated — no external state dependency.
  • Deterministic — same result every time if code hasn't changed.
  • Granular — pinpoints exactly where a bug exists.

The AAA Pattern

Structure every test using Arrange – Act – Assert:

  • Arrange — set up data and conditions; define variables, mock dependencies.
  • Act — execute the specific function under test.
  • Assert — verify the result matches expectation.

Test Coverage

A metric measuring how much code is executed when a test suite runs. Helps identify "dark" areas of the codebase.

TypeQuestion It Answers
Function CoverageHas each function been called?
Statement CoverageHas each executable line run at least once?
Branch (Decision) CoverageHas every path through a control structure been tested?
Condition CoverageHave all logical sub-expressions been evaluated as both true and false?
The Coverage Trap

100% coverage does not mean bug-free — it just means every line was executed. It doesn't prove logic is correct for all edge cases.

  • 90%+ — appropriate for critical business logic (e.g., payment processing).
  • 70–80% — often acceptable for UI glue code or low-risk scripts.
"Test coverage is a useful tool for finding untested parts of a codebase. It is of little use as a numeric target." — Martin Fowler

Test Driven Development (TDD)

Tests are written before production code.

TDD Cycle
  1. Write/edit tests for the next feature to be implemented.
  2. Run new tests → all new tests should fail.
  3. Write/edit production code.
  4. Run all tests → all tests should pass.
  5. Refactor code.
  6. Check test coverage; add tests if needed.
  7. Repeat for more features.

Merits of TDD: Clarifies and improves requirements. Produces higher-quality, well-structured code. Makes debugging easier. Improves readability through refactoring.

Use Case Based Testing

Links test cases back to requirements systematically:

  1. Prioritise use cases using a requirement-use case traceability matrix.
  2. Generate test scenarios from expanded use cases.
  3. Identify test cases from the scenarios.
  4. Generate test data (values).
  5. Generate test scripts.
Scenarios

A scenario is a concrete instance of a use case. Each use case has a primary scenario (the happy/normal path) and secondary scenarios (rare or exceptional cases). Both must be tested.

Integration Testing

Integration testing verifies the interactions and data exchange between components or modules. It assumes individual units have already been tested. Focus is on the correctness of the interface, not the modules themselves. Requires a module call graph.

Evaluation CriterionDescription
Fault localisationHow easy is it to pinpoint the source of a failure?
Effort neededHow much setup work (stubs, drivers) is required?
Degree of testingHow thoroughly are all modules exercised?
Parallel developmentCan testing begin before all modules are complete?

Integration Strategies

Big Bang

Non-incremental. Unit test each module in isolation, then integrate and test all at once.

  • Advantage: Convenient for small systems.
  • Disadvantage: Integration can only begin when all modules are ready. Fault localisation is very difficult.

Top-Down

Integrate moving downward in the control hierarchy from the main/top module. Uses stubs as placeholders for lower modules not yet implemented. Order can be depth-first or breadth-first.

  • Advantages: Fault localisation easier; few/no drivers needed; early prototype possible; major design flaws found first; flexible testing order.
  • Disadvantages: Requires many stubs/mock objects; low-level reusable modules may be inadequately tested.

Bottom-Up

Begins with atomic (leaf) modules, works upward. Uses drivers to test clusters of low-level components before integrating upward.

  • Advantages: Fault localisation easier; fewer stubs; operational modules tested thoroughly; can run parallel with implementation.
  • Disadvantages: Requires drivers; high-level modules (containing core solution logic) are tested last and least.

Sandwich (Hybrid)

Combines top-down and bottom-up. Top layer tested top-down while bottom layer tested bottom-up — they meet in the middle. Allows integration to begin early in the testing phase.

Regression Testing

When a module is added or changed, regression testing re-executes a subset of tests to ensure changes have not introduced unintended side effects. New code can establish new data flow paths, new I/O, or new control logic that breaks existing functionality.

A Good Regression Test Suite Contains
  1. A representative sample of tests that exercise all software functions.
  2. Tests focused on functions likely to be affected by the change.
  3. Tests focused on components that have been changed.
